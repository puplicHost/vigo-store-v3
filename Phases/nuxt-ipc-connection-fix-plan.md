# خطة حل خطأ `IPC connection closed` بشكل نهائي

> هذا الملف يضع خطة عملية لمعالجة خطأ `IPC connection closed` في `Nuxt + Vite` عند فتح `admin/dashboard`، مع التركيز على الوصول إلى حل نهائي وليس مجرد إعادة تشغيل مؤقتة.

---

## وصف المشكلة

الخطأ الظاهر هو:

```txt
Error: IPC connection closed
at Socket.onClose (...)
```

ويظهر أثناء طلب الصفحة:

- `GET /admin/dashboard`
- ثم ينتج عنه `500 Server Error`
- وبعدها يظهر `__nuxt_error`

هذا النوع من الأخطاء يعني غالبًا أن العملية الداخلية التي يعتمد عليها `Nuxt/Vite Node runtime` أغلقت أو انهارت أثناء تنفيذ الصفحة أو أثناء تحميل أحد الملفات المرتبطة بها.

---

## الهدف

الوصول إلى حل نهائي يضمن:

- توقف ظهور `IPC connection closed`
- فتح `admin/dashboard` بدون `500`
- معرفة السبب الحقيقي بدل علاج العرض فقط
- منع تكرار الخطأ أثناء التطوير لاحقًا

---

## التفسير الفني المختصر

خطأ `IPC connection closed` غالبًا ليس هو السبب الأساسي، بل هو **نتيجة** لانهيار شيء أعمق داخل بيئة التشغيل.

في مشاريع `Nuxt 3` يكون السبب الحقيقي عادة واحدًا من الآتي:

- خطأ runtime داخل ملف يتم تحميله في السيرفر
- import يسبب crash أثناء التقييم
- استخدام مكتبة متوافقة مع المتصفح فقط داخل SSR
- كود يعتمد على `window` أو `document` أو `localStorage` أثناء السيرفر
- exception داخل middleware أو composable أو layout
- crash في plugin
- مشكلة توافق في `node_modules`
- cache تالفة داخل `.nuxt` أو Vite
- مشكلة بيئية على ويندوز مع watcher أو child process

بالتالي، الحل النهائي يجب أن يبحث عن **أول خطأ فعلي** تسبب في سقوط عملية الـ IPC.

---

## الأعراض المرجحة في هذا المشروع

بما أن الخطأ يحدث عند الدخول إلى `admin/dashboard` تحديدًا، فالأماكن الأكثر احتمالًا هي:

- `app/pages/admin/dashboard.vue`
- `app/layouts/admin.vue`
- أي `middleware` خاص بالأدمن
- أي `plugin` يتم تحميله مع الأدمن
- أي `composable` أو `store` مستخدم في الـ dashboard
- استدعاء API أو ORM أو auth logic يُنفذ أثناء SSR

---

## أسباب محتملة مرتبة بالأولوية

### 1. كود متصفح يُنفذ على السيرفر

أمثلة شائعة:

- `window`
- `document`
- `localStorage`
- `sessionStorage`
- `navigator`

إذا استُخدمت مباشرة داخل:

- `setup`
- `useAsyncData`
- `computed`
- `plugin`
- `middleware`

فقد تؤدي إلى crash أثناء SSR.

### 2. import لمكتبة لا تدعم SSR

بعض المكتبات الخاصة بالرسوم البيانية أو المحررات أو الإشعارات أو الأنيميشن قد تعمل في العميل فقط، وإذا تم استيرادها مباشرة في السيرفر قد تكسر العملية.

### 3. exception غير معالج داخل `admin layout` أو `dashboard`

مثل:

- قراءة property من `undefined`
- نتيجة API غير متوقعة
- auth state غير جاهز
- destructuring على قيمة `null`

### 4. plugin أو middleware ينهار قبل تحميل الصفحة

خصوصًا إذا كان هناك:

- تحقق صلاحيات
- جلب بيانات المستخدم الحالي
- تهيئة theme
- تهيئة notification system

### 5. كاش Nuxt/Vite تالف

أحيانًا تكون المشكلة ناتجة عن ملفات build قديمة أو dependencies تغيرت مع بقاء cache قديم.

### 6. مشكلة توافق نسخة Node أو حزم المشروع

خاصة إذا:

- نسخة `Node.js` غير مناسبة لنسخة Nuxt الحالية
- تم تحديث بعض الحزم بدون تنظيف `node_modules`
- يوجد تعارض في `vite`, `nuxt`, `@nuxt/vite-builder`

## الخطة النهائية للحل

## المرحلة 1: عزل السبب الحقيقي

### 1.1 تحديد هل الانهيار من الصفحة أم من الـ layout

الهدف:

- معرفة هل الخطأ من `dashboard.vue` نفسه
- أم من `admin.vue`
- أم من middleware/plugin سابق للصفحة

خطوات التنفيذ:

1. استبدال محتوى `app/pages/admin/dashboard.vue` مؤقتًا بمحتوى بسيط جدًا.
2. إذا استمر الخطأ، ننتقل لفحص `app/layouts/admin.vue`.
3. إذا اختفى الخطأ، فالمشكلة داخل كود الـ dashboard نفسه أو ما يستدعيه.

### 1.2 تعطيل الأجزاء الحساسة تدريجيًا

داخل الصفحة أو الـ layout يتم تعطيل هذه الأجزاء مؤقتًا واحدًا واحدًا:

- استدعاءات `useAsyncData`
- استدعاءات API
- الرسوم البيانية
- الإحصائيات
- الإشعارات
- أي مكونات فرعية ثقيلة
- أي `watch` أو `computed` معقدة

الهدف هو الوصول إلى السطر أو المكون الذي يسبب سقوط العملية.

### 1.3 مراجعة الـ middleware والـ plugins

فحص الملفات التي تعمل قبل الصفحة:

- middleware الخاصة بالأدمن
- plugins العامة
- plugins العميل/السيرفر
- composables المستخدمة مبكرًا في الـ layout

أي خطأ هنا قد يظهر في الواجهة فقط على شكل `IPC connection closed`.

---

## المرحلة 2: إصلاح السبب البرمجي

### 2.1 حماية كود المتصفح

أي كود يعتمد على بيئة المتصفح يجب أن يُحمى باستخدام:

- `import.meta.client`
- `process.client`
- `onMounted`

أمثلة على الإصلاح:

```ts
if (import.meta.client) {
  const theme = localStorage.getItem('theme')
}
```

أو:

```ts
onMounted(() => {
  const theme = localStorage.getItem('theme')
})
```

### 2.2 عزل المكتبات غير المتوافقة مع SSR

إذا كانت الصفحة تستخدم مكتبات عميل فقط مثل:

- charts
- editors
- drag and drop libraries
- animation libraries

فيجب تطبيق واحد من الآتي:

- تحميلها داخل `client-only`
- نقل الاستيراد إلى client plugin
- استخدام `dynamic import`
- جعل المكون نفسه client-only

### 2.3 حماية النتائج القادمة من API أو auth

كل البيانات التي قد تكون:

- `null`
- `undefined`
- ناقصة

يجب التعامل معها بشكل defensive بدل الافتراض أنها موجودة دائمًا.

أمثلة:

- فحص القيم قبل destructuring
- استخدام fallback values
- عدم الوصول إلى nested properties مباشرة بدون guard

### 2.4 تقوية الـ middleware

أي middleware يجب أن:

- يرجع نتائج واضحة
- لا يرمي exception غير معالج
- يتعامل مع غياب session أو user أو token
- يسجل الأخطاء بوضوح أثناء التطوير

---

## المرحلة 3: تثبيت بيئة التشغيل

### 3.1 تنظيف ملفات البناء والكاش

بعد إصلاح السبب البرمجي يجب تنفيذ تنظيف كامل لـ:

- `.nuxt`
- `.output`
- `node_modules/.vite`

وإذا لزم الأمر:

- حذف `node_modules`
- إعادة تثبيت الحزم

هذه الخطوة مهمة لأن بعض أخطاء الـ IPC تستمر بسبب cache قديم حتى بعد إصلاح الكود.

### 3.2 تثبيت نسخة Node المناسبة

يجب التأكد من أن المشروع يعمل على نسخة Node مستقرة ومتوافقة مع Nuxt.

يفضل:

- توحيد النسخة عبر `.nvmrc` أو توثيقها داخل المشروع
- تجنب النسخ غير المستقرة أو غير المختبرة مع المشروع

### 3.3 مراجعة dependencies الحساسة

مراجعة توافق:

- `nuxt`
- `vite`
- `@nuxt/vite-builder`
- أي مكتبات SSR حساسة

إذا وُجد mismatch أو تحديث جزئي، يتم توحيد الحزم وإعادة التثبيت.

---

## المرحلة 4: منع تكرار المشكلة

### 4.1 إضافة قواعد واضحة للـ SSR

يجب اعتماد قاعدة داخل المشروع:

- أي شيء يعتمد على المتصفح لا يُنفذ في السيرفر
- أي مكتبة غير مضمونة التوافق توضع داخل `client-only`
- أي composable مشترك يجب أن يكون آمنًا في SSR

### 4.2 تحسين تسجيل الأخطاء

بدل أن نرى فقط `IPC connection closed`، نحتاج إظهار الخطأ الأصلي بسرعة عن طريق:

- logging أوضح داخل middleware
- logging أوضح داخل `useAsyncData`
- تغليف المقاطع الحساسة بـ `try/catch` عند الحاجة
- طباعة اسم المرحلة أو المكون الذي فشل

### 4.3 مراجعة صفحة `admin/dashboard` نفسها

إذا كانت الصفحة تجمع عدة مسؤوليات مرة واحدة، يفضل:

- تقسيمها إلى مكونات أصغر
- نقل المنطق الثقيل إلى composables واضحة
- جعل كل widget مستقلًا عن الآخر

هذا يسهل اكتشاف أي crash مستقبلًا ويمنع سقوط الصفحة بالكامل بسبب مكون واحد.

6. تنظيف cache وإعادة تشغيل البيئة

## تعريف الحل النهائي

يُعتبر الخطأ محلولًا بشكل نهائي عند تحقق الشروط التالية:

- `admin/dashboard` يفتح بدون `500`
- لا يظهر `IPC connection closed` بعد إعادة التشغيل
- لا يعود الخطأ بعد تعديل الملفات أو `HMR`
- السبب الحقيقي تم توثيقه داخل المشروع
- تم تأمين النقطة التي سببت الانهيار لمنع تكرارها

- تعديل plugin أو middleware يسبب crash
- نقل مكتبة إلى client-only
- تنظيف بيئة Nuxt/Vite وإعادة تثبيت dependencies
